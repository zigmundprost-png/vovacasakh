# Технический аудит vovacarlo.ru — план (read-only, production на VPS)

Production сайт https://www.vovacarlo.ru/ размещён на собственном VPS пользователя. Cloudflare-артефакты в репозитории (`@cloudflare/vite-plugin`, `wrangler.jsonc`) относятся к dev-стеку Lovable и НЕ означают, что боевой хостинг — Cloudflare. Реальный веб-сервер определяем по ответам с домена.

Аудит полностью read-only. Никаких изменений в VPS, DNS, SSL, firewall, systemd, Docker, файлах сайта, ENV, конфигах Nginx/Apache/Caddy, а также в коде репозитория — до отдельного одобрения. SSH на этом этапе не запрашиваем и не используем.

## Обнаруженный dev-стек репозитория (для контекста, не для выводов о проде)
TanStack Start 1.167 + React 19 + Vite 7 + Tailwind 4. Одна страница `src/routes/index.tsx` + `__root.tsx`, ~25 компонентов в `src/components/landing/*` (Hero, Works, Prices, Brief, Faq, MaxBriefDialog, MobileStickyCta и т.д.). Supabase-клиент присутствует, но лендинг статический.

## Что и как проверяем удалённо

1. **Идентификация VPS-инфраструктуры (только внешние сигналы)**
   - `curl -sSI` по `http://vovacarlo.ru`, `https://vovacarlo.ru`, `http://www.vovacarlo.ru`, `https://www.vovacarlo.ru`, глубокому пути и заведомо несуществующему пути.
   - Из заголовков вытащим: `Server`, `Via`, `X-Powered-By`, `CF-Ray`/`cf-cache-status` (для отсева Cloudflare-прокси), `Age`, `X-Cache`.
   - DNS: `dig +short A/AAAA/NS/CNAME`, `whois` по IP (ASN/владелец) — чтобы понять, стоит ли перед VPS CDN/прокси.
   - TLS: `openssl s_client -servername ... -connect ...:443 </dev/null` — issuer, срок действия, chain, поддержка TLS 1.2/1.3, ALPN (`h2`), OCSP stapling.
   - Редиректы: цепочки `http→https`, `www↔apex`, поведение `/index.html`, регистр URL, trailing slash.
   - Итог сведём в раздел «Фактическая серверная конфигурация VPS (по внешним признакам)».

2. **HTTP-заголовки и кэш (боевой домен)**
   - Security: HSTS, CSP, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, X-Frame-Options / CSP `frame-ancestors`.
   - Транспорт: `Content-Encoding` (gzip/br) по `Accept-Encoding: br, gzip` — отдельные запросы, сравнение размера.
   - Кэш: `Cache-Control`, `ETag`, `Last-Modified`, `immutable` для файлов с хешами в имени (`/assets/*-[hash].{js,css,woff2,webp}`), политика для `index.html`.
   - MIME: корректные `Content-Type` для `.js`, `.css`, `.woff2`, `.webp`, `.svg`, `.json`, `.xml`.
   - 404: реальный статус + наличие кастомной страницы, поведение при прямом открытии внутренних URL (SPA fallback на `index.html` или честный 404).
   - `/robots.txt`, `/sitemap.xml`, `/favicon.ico` — доступность и содержимое.

3. **Runtime в реальном браузере (Playwright в песочнице против https://www.vovacarlo.ru/)**
   - Chromium headless, viewport 1280×1800 + мобильные эмуляции.
   - Сбор: `console` (error/warning), network 4xx/5xx, mixed content, CORS, `unhandledrejection`, долгие таски.
   - Эмуляция Slow 3G + CPU 4× — что видит пользователь на медленной сети.
   - Скриншоты как доказательства к каждой проблеме.

4. **Функциональные сценарии (каждый — со скриншотом/логом как доказательством)**
   - Все ссылки MAX и WhatsApp: `href`, схема, `target`, `rel="noopener"`; `tel:` ссылки.
   - Якорная навигация из шапки, скролл к секциям.
   - Галерея Works: фильтры «Все/Ремонт/Сборка», открытие/закрытие фото, Escape в модалках, свайпы на touch.
   - FAQ: раскрытие, `aria-expanded`/`aria-controls`.
   - «Показать ещё» в Works.
   - Конструктор заявки `Brief` + `MaxBriefDialog`: заполнение, копирование текста, поведение при недоступном `navigator.clipboard` (эмуляция `Object.defineProperty(navigator, 'clipboard', { value: undefined })`), открытие MAX после заявки, защита от двойных кликов (быстрая серия кликов — считаем срабатывания).

5. **Адаптив 320/360/375/390/414/768/1024/1280/1440/1920 + portrait/landscape для мобильных**
   - Горизонтальный скролл (`scrollWidth > innerWidth`).
   - Перекрытия sticky/fixed CTA (`MobileStickyCta`, `DesktopFloatingCta`) с контентом и футером.
   - Safe-area (iPhone notch, эмуляция `viewport-fit=cover`).
   - Длинные русские слова: overflow, переносы.
   - Тач-жесты в галерее и модалках.

6. **Lighthouse (боевой URL, отдельно mobile и desktop)**
   - `lighthouse https://www.vovacarlo.ru/ --preset=desktop --output=json,html` и без пресета для mobile.
   - Метрики: Performance, Accessibility, Best Practices, SEO + LCP, INP/TBT, CLS, FCP, Speed Index — свести в таблицу.

7. **Изображения (из HAR боевого сайта)**
   - Количество и вес картинок при первом экране, что грузится ниже фолда.
   - `loading="lazy"`, явные `width/height`, `srcset`/`sizes`.
   - intrinsic vs displayed size (кандидаты «полноразмер вместо thumb», особенно в `Works.tsx`).
   - PNG/JPG-кандидаты на WebP/AVIF (без автоматической конверсии — только рекомендация).

8. **Доступность WCAG 2.2 AA**
   - `@axe-core/playwright` прогон + ручные проверки: tab-порядок, видимый focus, focus-trap и возврат фокуса (`MaxBriefDialog`, лайтбокс фото), `aria-label` иконочных кнопок, `aria-expanded/controls` в `Faq.tsx`, alt у всех `<img>`, контраст, семантика `<button>`/`<a>`, `<label>` полей `Brief`, сообщения об ошибках, `prefers-reduced-motion`.

9. **SEO и метаданные**
   - `head()` в `__root.tsx` и `index.tsx`: title, description, canonical, OG/Twitter, `lang="ru"`, favicon, JSON-LD `LocalBusiness` (адрес, телефон, часы, гео Южно-Сахалинска), структура H1–H3, отсутствие дублей.
   - Соответствие `robots.txt` и `sitemap.xml` реальным маршрутам.

10. **Кросс-браузер (smoke, без правок в репо)**
    - Прогон боевого URL через Playwright: Chromium, Firefox, WebKit + эмуляции Mobile Chrome (Pixel 5) и Mobile Safari (iPhone 13). Только чтение — тестовые файлы в репозиторий не пишутся.

## Формат отчёта

Единая таблица находок:

```text
Категория | Приоритет | Проблема | Где найдена | Как воспроизвести (команда/шаги) | Доказательство | Причина | Рекомендация | Риск изменения
```

`Категория` — одно из:
- **Подтверждённая ошибка** (есть воспроизводимое доказательство сбоя пользовательского сценария).
- **Риск** (поведение в норме, но конфигурация или код создаёт потенциальный сбой в реальных условиях).
- **Рекомендация по улучшению** (необязательная оптимизация — НЕ ошибка).
- **Ограничение проверки** (что не удалось проверить без SSH/приватного доступа).

`Приоритет` P0/P1/P2/P3 присваивается только «ошибкам» и «рискам»; «рекомендации» идут без P0/P1.

## Обязательные разделы в конце отчёта

1. Явное утверждение «Production размещён на VPS пользователя», без Cloudflare-предположений.
2. **Фактическая серверная конфигурация VPS (по внешним признакам)** — что удалось определить: веб-сервер (Nginx/Apache/Caddy/другое) по `Server`-заголовку и поведению, наличие/отсутствие прокси/CDN перед ним, схема редиректов http→https и www↔apex, издатель и срок SSL, ALPN/HTTP2, компрессия, кэш-политика, обработка 404 и прямых URL, фактически возвращаемые security-заголовки.
3. **Что проверено удалённо** — список выполненных проверок с использованными командами.
4. **Что требует SSH-доступа** — список пунктов, которые невозможно достоверно подтвердить снаружи (реальные конфиги Nginx/Apache/Caddy, права на файлы, systemd-юниты, версия и модули веб-сервера, содержимое webroot, cron/logrotate, версии PHP/Node на VPS, точная схема Brotli/gzip-настроек, TLS-конфиг, fail2ban/UFW).
5. **Безопасные команды для будущего SSH-этапа** — перечень строго read-only команд (например: `nginx -T`, `systemctl status nginx`, `ss -tlnp`, `ls -la <webroot>`, `stat`, `cat` конфигов, `journalctl -u nginx --since ...`, `curl -I` изнутри), с явной пометкой «выполнять только после отдельного одобрения».
6. Таблица Lighthouse mobile/desktop со всеми ключевыми метриками.
7. Отдельные списки P0 и P1 (только из категорий «Подтверждённая ошибка» и «Риск»).
8. Список файлов репозитория, которые потенциально предстоит менять при последующем фиксе (без внесения изменений сейчас).
9. Рекомендуемый порядок исправлений (после отдельного одобрения).

## Явные ограничения этого аудита

- Не запускаю ничего на VPS, не подключаюсь по SSH, не трогаю DNS/SSL/firewall/Docker/systemd.
- Не меняю файлы репозитория, `vite.config`, зависимости, тексты, цены, фото, бизнес-логику.
- Не создаю Playwright-конфиг и тестовые файлы в репозитории; прогоны браузера выполняются как временные скрипты в песочнице только для сбора доказательств.
- Отсутствие необязательной оптимизации не помечается как ошибка — только как «Рекомендация по улучшению».
- Каждая «Подтверждённая ошибка» сопровождается воспроизводимой командой/шагами и артефактом (скриншот, лог, HAR-фрагмент, ответ curl).

После одобрения плана выполняю шаги 1–10 и возвращаюсь с полным отчётом. Исправления — отдельным этапом после отдельного одобрения.
